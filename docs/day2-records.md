# Day 2: Data Carriers — Records

In Java 8, if you wanted to create a simple class just to hold data (like a Data Transfer Object, or DTO, in Spring Boot), you had to write a massive amount of boilerplate code. You needed `private final` fields, a constructor, getters, `equals()`, `hashCode()`, and `toString()`. Tools like Lombok became heavily relied upon just to hide this mess.

Introduced officially in Java 16, **Records** solve this at the language level. A Record is a transparent, immutable carrier for your data.

---

## The Java 8 Way — 30 Lines of Boilerplate

```java
static class UserPojo {
    private final String name;
    private final int age;

    public UserPojo(String name, int age) {
        this.name = name;
        this.age = age;
    }

    public String getName() { return name; }
    public int getAge() { return age; }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        UserPojo userPojo = (UserPojo) o;
        return age == userPojo.age && Objects.equals(name, userPojo.name);
    }

    @Override
    public int hashCode() {
        return Objects.hash(name, age);
    }

    @Override
    public String toString() {
        return "UserPojo{name='" + name + "', age=" + age + "}";
    }
}
```

Every field means more constructor parameters, more getters, more lines in `equals()` and `hashCode()`. Miss one and you have a subtle bug.

## The Modern Way — One Line

```java
public record UserRecord(String name, int age) {}
```

That single line gives you the constructor, accessors, `equals()`, `hashCode()`, and `toString()`. All auto-generated by the compiler.

---

## Key Features of Records

### 1. Zero Boilerplate

You declare the state in the header — `String name, int age` — and the compiler does the rest:

```java
var user = new UserRecord("Alice", 28);

user.name();          // "Alice" — accessor generated
user.toString();      // "UserRecord[name=Alice, age=28]"
user.equals(other);   // structural equality by all fields
user.hashCode();      // derived from all fields
```

### 2. Immutability

All fields in a record are implicitly `final`. There are no setters. Once created, a record cannot be changed. This makes records inherently thread-safe and safe to use as map keys or set elements.

### 3. Accessor Naming — No `get` Prefix

Unlike traditional POJOs, record accessors drop the `get` prefix:

```java
// POJO
pojo.getName()   // "get" prefix

// Record
record.name()    // just the field name
```

This is a deliberate design choice. Records are not beans — they're named tuples with behavior. The shorter name reads more naturally, especially when chained.

### 4. Local Records

You can declare a record inside a method to quickly group data temporarily — something that wasn't practical with traditional classes:

```java
public static void main(String[] args) {
    record Point(int x, int y) {}
    var p = new Point(10, 20);
    System.out.println(p);  // Point[x=10, y=20]
}
```

This is useful when a method needs to return or intermediate a pair/triple of values without polluting the class namespace.

## Customizing Records

Records are rigid by default, but you can customize them when needed.

### Compact Constructor (validation)

```java
public record Email(String address) {
    public Email {
        if (address == null || !address.contains("@")) {
            throw new IllegalArgumentException("Invalid email");
        }
    }
}
```

The compact constructor strips the parameter list — it's just the body. The assignment to `this.address` happens automatically after your code runs.

### Adding Methods and Static Fields

```java
public record Money(int amount, String currency) {

    // Static field — allowed (instance fields are not)
    public static final Money ZERO = new Money(0, "USD");

    // Instance method — allowed
    public Money add(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException("Currency mismatch");
        }
        return new Money(this.amount + other.amount, this.currency);
    }
}
```

### What you CANNOT do

- Add instance fields (the state is exactly what's in the header)
- Extend a class (records implicitly extend `java.lang.Record`)
- Be mutable (fields are always `final`)

## Records vs Other Options

| Feature | POJO | Lombok `@Data` | Record |
|---|---|---|---|
| Boilerplate | High | None (generated) | None (language-level) |
| Immutability | Manual | Manual | Built-in |
| `equals`/`hashCode`/`toString` | Hand-write | Generated | Generated |
| Dependency | None | Lombok lib | None (JDK 16+) |
| Instance fields | Any | Any | Header only |
| Extends a class | Yes | Yes | No |
| Can be mutable | Yes | Yes | No |
| IDE support | Full | Requires plugin | Full |

---

## When to Use Records

- **DTOs and API responses** — immutable data traveling between layers
- **Method return types** — returning multiple values without a full class
- **Keys in maps** — correct `equals`/`hashCode` for free
- **Configuration objects** — small, immutable settings holders
- **Pattern matching targets** — records work beautifully with `instanceof` pattern matching (Day 3)

## When NOT to Use Records

- You need mutability (builders, stateful objects)
- You need to extend a base class
- You need non-header instance fields
- You're targeting frameworks that require JavaBeans naming (`getXxx`) — though many modern frameworks now support records
